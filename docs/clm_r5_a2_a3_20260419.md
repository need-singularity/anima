## 1. r5-a2 — mmap-equivalent corpus loader

### Status: completed

The mmap-equivalent byte-window loader was authored + 14/14 self-test
verified on 2026-04-16 (pre-r5). Today's audit confirmed the module is
production-ready; the r5-a2 box turned into a compliance check rather
than new code.

| field                | value                                                    |
|----------------------|----------------------------------------------------------|
| standalone module    | `training/clm_mmap_loader.hexa` (335 LOC)                |
| inlined subset       | `training/train_clm.hexa` L36-81 (corpus_open + window)  |
| self-test            | 14/14 PASS (`clm_mmap_loader_self_test`, 2026-04-16)     |
| inline rationale     | pod2 scaffold transpiles a fixed DEP list — inlining avoids the silent-exit-on-imports trap (`feedback_hexa_silent_exit_5_imports`) |
| disk path            | `dd if=<corpus> bs=1 skip=<off> count=<n> ... \| od -An -vtu1` — O(1) syscalls per window, page-cache friendly |
| resident RAM         | ~32 KiB / step at batch=8 seq=512 (vs 5.38 GB for whole-file load, vs 172 GB hexa_array blow-up pre-mmap) |
| native mmap(2) FFI   | deferred — hexa C-shim not strictly needed; swap-in path documented in loader header |

Smoke on hetzner with `/root/Dev/anima/data/corpus.txt` (5.24 MB):

```
[clm_mmap_loader] opened /root/Dev/anima/data/corpus.txt len=5243172 bytes (~5 MiB)
[load_corpus] path=/root/Dev/anima/data/corpus.txt n_tokens=5243172 block_size=128
```

Appears twice per training run because `load_corpus` + `_current_corpus_desc`
both stat — a `wc -c` no-op per call, O(1).

## 2. r5-a3 — `--config` JSON + `--resume` wiring

### Status: completed

Instead of authoring a separate `training/deploy/clm_r5_launch.hexa`
wrapper (would triple the dep-bundle footprint), the two flags land
directly on `train_clm.hexa` so one AOT binary covers both smoke and
pod fire.

### 2.1 `--config <path>` — JSON scalar override

Added to `train_clm.hexa` L3255+ (CLI dispatch section):

- `_cfg_json_read_scalar(text, key)` — mirrors `clm_val.hexa:json_read_scalar`
  pattern, returns the raw value as `string` (handles `"key":"val"` and
  `"key":123`). Not a full parser — project-emitted JSON only.
- `_cfg_json_read_int(text, key, fallback)` and `_cfg_json_read_float`
  wrappers.
- Applied inside the train-mode block after the `--scale` default is
  materialised. JSON fields **override** the `--scale` template; if a
  field is missing from JSON the template value passes through
  untouched.

Fields mapped (flat, not nested — hexa JSON reader scans the file linearly):

| JSON key              | → ScaleConfig field   |
|-----------------------|-----------------------|
| `model_tag`           | `name`                |
| `d_model`             | `d_model`             |
| `n_layer`             | `n_layer`             |
| `n_head`              | `n_head`              |
| `n_kv_head`           | `n_kv_head`           |
| `block_size`          | `block_size`          |
| `consciousness_dim`   | `consciousness_dim`   |
| `atoms`               | `atoms`               |
| `cells_per_atom`      | `cells_per_atom`      |
| `batch_size`          | `batch_size`          |
| `lr_max`              | `lr`                  |
| `steps`               | `_total_steps` (only if CLI default literal) |
| `save_every`          | `_save_every` (only if CLI default literal) |

Precedence: explicit CLI flag > JSON config > `--scale` template.

### 2.2 `--resume`

Added a separate helper plus flag handling:

- `_resume_find_last_step(ckpt_dir: string) -> int` — shell-driven
  (`ls <dir>/step_* | sed -n 's#.*/step_\([0-9]\+\).*#\1#p' | sort -n | tail -1`),
  returns `-1` on missing dir.
- When `--resume` is present with `--ckpt-dir`, the binary scans and
  logs the highest `step_N` directory found. **Counter-only resume
  for this round** — full weight restore from `HEXACKPT-v1` requires
  a matching `load_checkpoint()` decoder that is not yet wired
  (tracked as r5-a3.5 before pod fire).
