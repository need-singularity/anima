### Root cause

`hexa-lang/self/native/hxblas_linux.c` wraps `hxblas_sgemm` and `hxblas_attn_softmax_causal` inside a `#ifdef __linux__` / `#endif /* __linux__ */` block (lines 38–315). On macOS the built `libhxblas.dylib` (33696 bytes, 2026-04-17) only exports:

```
hxblas_bf16_to_f32
hxblas_bf16_version
hxblas_bgemm
hxblas_bgemv
hxblas_f32_to_bf16
hxblas_matmul_scalar
hxblas_version
```

The training binary's top-level init (line 3304–3305 of the codegen) dlopens `libhxblas.dylib` and dlsyms `hxblas_sgemm` + `hxblas_attn_softmax_causal`, which resolve to NULL on Mac. The top-level then unconditionally calls `smoke_train_step()` → `_ce_fwd_bwd` → the NULL fn ptr → SIGSEGV, before the `if (_train_mode)` CLI branch ever runs.

`libhxlmhead.dylib` is fine on Mac (`hxlmhead_bwd`, `hxlmhead_version` resolve OK — per C7 libhxlmhead confirmed).

### Fallback paths tried

- Path A (run existing `/tmp/clm.bin`): same code path, same segfault — stale binary from 22:48 already dlopened hxblas in identical manner.
- Path B (symlink `training/build/libhxblas.dylib` → hexa-lang build): symlink did resolve dlopen but the symbols still missing (same library, same exports). No effect.
- Path C (rebuild libhxblas.dylib with Accelerate framework): no-op — `#ifdef __linux__` excludes the sgemm path regardless of flags.
- Path D (patch `hxblas_linux.c` macOS branch to add `hxblas_sgemm` wrapping `cblas_sgemm$NEWLAPACK`): **out of scope** — modifying the hexa-lang ABI from an anima-side smoke is risky, should be a separate hexa-lang commit with xverify.
