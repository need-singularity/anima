## Root finding

**No real per-persona LoRA training pipeline exists** in the repo. The three "LoRA" files:

1. `training/persona_lora_train.hexa` — in-process arithmetic on random vectors, classifies synthetic persona centroids via nearest-residual. Produces a `H100TrainPlan` struct but never executes anything.
2. `training/train_alm_lora.hexa` — toy 2-layer decoder LoRA on `nn_core` primitives, same CPU-only hexa runtime.
3. `training/train_alm_14b.hexa` — real Qwen2.5-14B full LoRA trainer. One global LoRA adapter. No per-persona adapter loop, no persona-labeled corpus consumer, no per-persona ckpt naming.

The anima-train.json TALM-P2-1 `done_criteria` says "6 LoRA × 2000 step, eval persona 일관성 ≥ 0.85". Executing this requires **writing** a new training harness that:

- Loads Qwen2.5-14B (or the r4 LoRA merge) as base
- Iterates 6 persona-labeled corpus shards
- Trains 6 independent LoRA adapters (rank=16, alpha=32, dropout=0.05 per `plan_h100_train()`)
- Saves 6 adapters to `anima-models/alm14b/r4/lora-persona/{persona_tag}/`
- Eval harness measures per-persona consistency (classification or generation-based)

None of these components exist. Even the smoke scope (1 persona × 500 step) requires writing the harness.

## Decision

**Do not launch a pod.** Budget protection rule (2h/$10 cap, smoke-only) and existing task constraint ("base model 로딩 실패 → abort"): here the failure is upstream — the *loader does not exist*.

Alternatives considered and rejected:

- **Launch pod, write harness on pod in real time**: violates task constraint "smoke 만 이 에이전트" and would consume far more than 2h/$10 (pipeline authoring + debug typically 4-8h for new harness).
- **Reuse train_alm_14b.hexa as-is, single global LoRA on persona_0 corpus only**: silently mislabels as TALM-P2-1 smoke when it's actually TALM-P1-1 redux (already done). Misleading metric.
- **Launch pod to load base + idle**: pure spend, no progress signal.
